# Mapsi 발표 자료 전달 문서 (PPT 제작자용)

> 본 문서는 Mapsi 프로젝트 발표용 PPT 를 제작하는 담당자에게 전달하기
> 위한 원고이다. 섹션별로 ① 슬라이드에 박을 짧은 카피 ② 발표자/제작자용
> 보조 설명 ③ 시각 요소 제안 세 층으로 구성된다. 불필요한 과장 없이
> 실제 프로젝트 수치·파일명·ADR 번호를 그대로 기재한다.
>
> - 프로젝트 한 줄: "Markdown 문서를 한/글(HWPX) 로 변환하면서 각 단락에
>   적절한 스타일을 자동으로 입히는 변환기" — 이름 **Mapsi** 는
>   "맵시(모양새·차림새)" 에서 따옴.
> - 예상 시연 소요: 약 2–3 분. 전체 발표는 10–12 분 분량으로 설계.

---

## 1) 프로젝트 목표 (1–2 슬라이드)

**슬라이드 카피:**

- 제목: **"마크다운을 한/글 문서로 — 스타일까지 자동으로"**
- 부제: Markdown Adapter for Paragraph-Style Injection

**핵심 문제 3 줄:**

1. 연구·보고 문서는 Git · Markdown 으로 쓰기 편한데, 제출·협업은 한/글(.hwpx)
2. 기존 변환기는 본문만 옮기고 **스타일은 사람이 일일이 지정**해야 함
3. "구조(마크다운) ↔ 스타일(한/글)" 매핑을 **데이터로 명시**하고 자동화

**프로젝트가 하는 일 (1 줄씩):**

- 마크다운의 **역할**(제목, 본문, 리스트, 인용, 코드, 표, 그림, 각주,
  수식, 하이퍼링크) 을 한/글의 **스타일 이름**에 매핑
- 한/글이 발급한 **`header.xml`** 을 진실원으로 두고, 이름으로 조인해
  `paraPrIDRef` / `charPrIDRef` 를 본문에 박음
- **시각 속성**(폰트·색·여백) 은 건드리지 않음 → 한/글 스타일 편집기에서
  한 번에 조정 가능

**시각 요소:** before/after 비교 이미지 1 장 — 좌측 `.md`, 우측 한/글에서
열린 `.hwpx` (본문·제목·코드 블록이 서로 다른 스타일로 자동 착용된 장면).

## 2) 설계 (2–3 슬라이드)

### 2-1. 핵심 설계 원칙 — "진실원 분리"

**슬라이드 카피 (다이어그램용):**

```text
마크다운 토큰 ─ 역할(role) ──── spec/styles.yaml ──── 스타일 이름
                                                        │
                                                        ▼ (이름으로 조인)
                                           templates/Contents/header.xml
                                               (한/글이 발급한 스타일 정의)
                                                        │
                                                        ▼
                                   paraPrIDRef · charPrIDRef · styleIDRef
                                                        │
                                                        ▼
                                          Contents/section0.xml (본문 XML)
```

**메시지 (한 줄씩):**

- `spec/styles.yaml` — **정책** (어떤 역할을 어떤 스타일 이름에 보낼지)
- `templates/Contents/header.xml` — **사실** (한/글이 해당 이름에 배정한
  ID 가 무엇인지)
- 빌더는 정책·사실을 **이름으로 조인**해서 하드코딩 없이 동작
- ID 가 바뀌어도 코드는 변경 없이 그대로 동작 (유지보수성)

### 2-2. 5 단계 파이프라인

1. **Parse** (`mapsi/parser.py`) — markdown-it-py 로 토큰화 → Block 객체
2. **Map** (`spec/styles.yaml`) — 역할 → 한/글 스타일 이름
3. **Build** (`mapsi/builder/`) — `header.xml` 룩업 → 본문 XML 요소 생성
4. **Pack** (`mapsi/packaging/`) — OPF 매니페스트 + ZIP 패키징으로 `.hwpx`
5. **Inspect** (`mapsi/inspect.py`) — 한/글 없이 셸에서 구조적 정확성 확인

**시각 요소:** 위 다이어그램을 5 박스 플로우로 그림. 각 박스 아래 해당
모듈 이름 병기.

### 2-3. 아키텍처 결정 (ADR)

**슬라이드 카피:** 5 개 ADR 이 프로젝트의 **"왜 이렇게 했나"** 를
기록한다.

| #    | 주제          | 결정 요지                                                                            |
| ---- | ------------- | ------------------------------------------------------------------------------------ |
| 0001 | 표 캡션 승격  | 표 직전 문단이 "표 N." 로 시작하면 `table_caption` 역할로 자동 승격                  |
| 0002 | 수식 마커 모드| `hp:equation` XML 대신 평문 마커(`[hnc 수식]…[/hnc 수식]`) 로 삽입해 한/글 수식 편집기가 파싱 |
| 0003 | Team C 통합   | 이미지·매니페스트·패키징 구현을 cherry-pick 합성 머지로 흡수                         |
| 0004 | 인라인 서식   | bold/italic/code/strike/link 에 대한 charPr ID 25–30 정책. 링크는 v0.1.2 한/글 정식 스펙 |
| 0005 | CP4 통합 골든 | 학술 논문 길이 픽스처 `cp4_full` 로 회귀 커버                                         |

## 3) 개발 과정 (1–2 슬라이드)

**페이즈별 진행 (발표에서는 3–5 개 핵심만):**

- **Phase 0** — HWPX 포맷 탐사·역공학 (샘플 HWPX 분석, `header.xml` 구조 파악)
- **Phase 1–3** — 파서·빌더·패키지 골격: 본문/제목/리스트
- **Phase 4–9** — 인용·코드·표·그림·각주·참고문헌·수식 (ADR 0001, 0002 적립)
- **Phase 10** — 인라인 서식 (ADR 0004): bold, italic, code_inline, strike,
  **hyperlink**
- **Phase 11** — 통합 골든 `cp4_full` (ADR 0005): 모든 요소가 한 문서에서
  상호작용하는 회귀 방어선
- **UI Phase** — Streamlit 브라우저 인터페이스 (파일 업로드 / 텍스트 붙여넣기
  / ZIP 업로드 3 탭)

**협업 방식:**

- 3 인 역할 분담 (B: 파서·빌더 / C: 이미지·매니페스트·패키징 / UI 담당자:
  Streamlit)
- Git flow: `feature/*` 브랜치 → PR → main. ADR 로 의사결정 남김.
- 핸드오프 문서 (`docs/c_handoff.md`, `spec/interfaces.md`) 로 인터페이스
  고정

**테스트 전략 (중요, 슬라이드 1 장 가치):**

- **362 개 단위/통합/골든 테스트** (현재 모두 통과)
- **골든 회귀 픽스처 11 개**: `01_headings` → `10_inline` + `cp4_full`
- 각 픽스처마다 `.md`(입력) 과 기대 블록 시퀀스(`expected.json`) 를 함께
  버전관리
- "한/글이 발급한 ID 가 바뀌어도 코드는 무사" 를 보장하는 **이름 기반
  룩업** 회귀
- 수식 경로는 CI 에서 `MAPSI_NO_LLM=1` 강제 → LLM 의존성 없이 회귀 안정

## 4) 결과 — 각 요소별 처리 (메인 섹션, 3–5 슬라이드)

각 요소별로 **입력(.md) → 출력(한/글 화면)** 의 **한 쌍** 이미지를
넣으면 발표 효과가 가장 큼. 아래는 요소별 한 줄 설명 + 강조 포인트.

| #  | 요소             | 마크다운 표기                     | 한/글 처리 결과                    | 강조 포인트                                    |
| -- | ---------------- | --------------------------------- | ---------------------------------- | ---------------------------------------------- |
| 1  | **제목**         | `#` ~ `######`                    | 개요 1~6 스타일 + 개요 번호 자동   | 6 레벨까지 트리 자동 생성                      |
| 2  | **본문 문단**    | 일반 문단                         | "본문" 스타일                      | —                                              |
| 3  | **순서 없는 목록**| `- item`                          | "네모/동그라미/줄" 스타일 (3 depth)| 들여쓰기 깊이별 자동 분기                      |
| 4  | **순서 있는 목록**| `1. item`                         | "번호1/2/3" 스타일                 | 원문 번호 보존 (자동번호 아님)                 |
| 5  | **인용문**       | `> quote`                         | "인용" 스타일 + 인라인 마크 보존   | 인용 내부 bold/link 도 살아남음                |
| 6  | **코드 블록**    | ` ```…``` `                       | "코드" 스타일 (고정폭 글꼴)        | 줄바꿈·들여쓰기 그대로                         |
| 7  | **표**           | `\| a \| b \|`                    | 표 셀 + 자동 캡션 승격             | "표 N." 문단 자동 인식 (ADR 0001)              |
| 8  | **그림**         | `![alt](path)`                    | HWPX 이미지 embed + 캡션           | 누락 시 placeholder 자동 대체                  |
| 9  | **각주**         | `[^1]` + `[^1]:`                  | `hp:footNote` + "각주" 스타일      | 번호 자동 관리                                 |
| 10 | **참고문헌**     | 제목 "참고문헌" 뒤 문단들         | "참고문헌" 스타일 자동 적용        | 블록 휴리스틱으로 승격                         |
| 11 | **수식**         | `$…$` / `$$…$$`                   | 평문 마커 `[hnc 수식]…[/hnc 수식]` | ADR 0002. LLM 있으면 HNC 스크립트, 없으면 LaTeX 원문 |
| 12 | **인라인 서식**  | `**bold**`, `*it*`, `` `code` ``  | charPr 25–29 매핑                  | 조합은 Degradation Policy                      |
| 13 | **하이퍼링크**   | `[text](url)` · 베어 URL · 이메일 | `hp:fieldBegin HYPERLINK` 정식 필드 (charPr 30, 파란 + underline) | **한/글에서 실제 클릭 동작** (v0.1.2) |

### 4-별도 강조 슬라이드 1 장: "하이퍼링크 — 실제 클릭 가능하기까지"

이게 이번 프로젝트의 **가장 교훈적인 사례** 라 별도 슬라이드 추천.

- 1 차 구현 (v0.1.1): 오픈소스 `python-hwpx` 의 `add_hyperlink` 형태 차용
  → **한/글에서 클릭 활성화 안 됨**
- 원인 역공학: 한/글이 직접 저장한 HWPX 를 풀어 구조 비교 →
  `<hp:parameters>` 블록(Command / Path / Category / TargetType /
  DocOpenType / ToolTip 7 개) 이 필수
- 2 차 구현 (v0.1.2): 정식 스펙으로 교체 + `fieldid` 페어링 + `fieldEnd`
  뒤 빈 `<hp:t/>` → **뷰어에서도 클릭 동작**
- 교훈: **"스펙 문서" 가 아닌 "실제 구현 동작" 을 진실원으로 삼아야**
  한다

**시각 요소:** v0.1.1 vs v0.1.2 XML diff 2 컬럼 + 우측에 "클릭 안 됨 →
클릭 됨" 스크린샷.

### 4-별도 강조 슬라이드 1 장: "CP4 통합 골든"

- 학술 논문 길이의 실제적 마크다운 한 벌 (`tests/golden/cp4_full/input.md`)
- 13 개 요소가 **동시에** 등장하는 단일 문서로 상호작용 회귀 방어
- 이 픽스처 하나가 단위 테스트로는 못 잡는 "표 직후에 그림" 같은 조합
  문제를 잡음

**시각 요소:** cp4_full 로 생성된 `.hwpx` 를 한/글에서 연 전체 문서
스크린샷 (페이지 여러 장 썸네일).

## 5) 추후 개선사항 (1 슬라이드)

우선순위 높은 것부터.

**기능**

- **문단 간 수직 여백** — 현재 `paraPr` margin 이 0 이라 헤딩이 본문에
  바짝 붙음. 블록 경계에 빈 `<hp:p>` 삽입하는 규칙 도입
  (`header.xml` 불가침 제약하에서의 해법)
- **내부 앵커 점프** — `[링크](#section)` 가 현재 URL 만 보존. v0.3
  북마크(`hp:bookmark`) 연결 구현 예정 (ADR 0004 deferred 항목)
- **표 내 인라인 서식** — 셀 안의 bold/link 도 charPr 반영
- **이미지 자동 리사이즈** — 업로드된 원본이 페이지 폭을 초과할 때 비율
  유지 축소

**품질/운영**

- **Streamlit UI 테스트** — 현재 UI 는 수동 테스트만. playwright 자동화
- **CI 통합** — GitHub Actions 에 `pytest` + `ruff` + `mypy` 파이프라인
- **로컬 LLM 옵션** — `ollama` 지원으로 API 키 없이 수식 HNC 변환

**확장**

- **Google Docs / Word(.docx) 입력** — 마크다운 이외의 소스도 지원
- **플러그인 스타일 카탈로그** — 논문/보고서/이력서 등 템플릿별
  `styles.yaml` 제공

## 6) 시연 (1 슬라이드 + 영상 or 라이브)

### 6-1. 시연 시나리오 (2 분 분량)

**씬 1 — 입력 (15 초)**

- 터미널에 `tests/golden/cp4_full/input.md` 를 `bat` 또는 에디터로 띄워
  화면에 보여줌
- 제목·본문·리스트·표·그림·하이퍼링크가 한 문서에 있는 것을 빠르게 스크롤

**씬 2 — Streamlit UI 변환 (40 초)**

- `streamlit run streamlit_app.py` 실행 화면 (이미 실행 중일 것)
- **"Markdown + 이미지 ZIP"** 탭에서 `tests/input_sample.zip` 업로드
- "HWPX 다운로드" 클릭 → 생성된 파일 다운로드

**씬 3 — 한/글에서 열기 (50 초)**

- 다운로드한 `.hwpx` 를 한/글에서 엶
- 스크롤하며 제목·본문·리스트·표·그림이 각각 **다른 스타일로 자동 적용**된
  것 확인
- **하이퍼링크 클릭** → 기본 브라우저로 이동 (하이라이트 포인트)
- 한/글 **스타일 편집기**에서 "본문" 스타일의 글꼴/색을 바꾸면 → 전체
  본문이 한 번에 바뀜 → **"스타일 자동 부여의 실용 효과"** 시연

**씬 4 — 검증 (15 초)**

- 터미널에서 `python -m mapsi.inspect <다운받은파일>.hwpx --styles` 실행
- "한/글 없이도 구조적 정확성 확인 가능" 포인트

### 6-2. 영상 촬영 시 주의

- 해상도: 1080p 이상 (코드 가독성)
- 자막: 각 씬 시작 시 "1) 입력 마크다운", "2) UI 변환",
  "3) 한/글에서 확인", "4) 구조 검증"
- 한/글 화면 녹화 시 테마를 밝은 테마로 (다크 + 한국어 폰트 렌더링 깨짐
  주의)

### 6-3. 백업 (영상 불가 시 캡처 스틸)

필수 스틸 4 장:

1. `.md` 원본 편집기
2. Streamlit UI 변환 화면
3. 한/글에서 연 결과 (가장 중요, 여러 장 가능)
4. `mapsi.inspect` 터미널 출력
