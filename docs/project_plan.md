# Mapsi 프로젝트 계획

**Markdown Adapter for Paragraph-Style Injection**

마크다운 문서를 한/글 HWPX 로 변환하면서 각 단락에 적절한 hwpx 스타일을 자동으로 부여하는 변환기이다. "맵시" 는 "모양새, 차림새" 를 뜻하는 한국어이며, 본 프로젝트는 마크다운의 구조를 hwpx 의 스타일로 맵시 있게 옮기는 도구이다.

---

## 프로젝트 개요

### 배경

한국어 학술 및 공문서 작성 환경에서 한/글의 HWPX 포맷은 사실상의 표준이지만, 초안 작성 단계에서는 마크다운이 훨씬 빠르고 버전 관리가 용이하다. 따라서 "마크다운으로 초고를 작성한 뒤 HWPX 로 변환해 한/글에서 최종 편집" 하는 워크플로가 실용적이다. 현재 Pandoc 을 포함해 어떤 오픈소스 도구도 HWPX 라이터를 제공하지 않으며, 본 프로젝트는 그 공백을 메우는 것을 목표로 한다.

### 핵심 목적

본 프로젝트의 목표는 "마크다운 문서의 구조적 역할을 한/글의 스타일 시스템에 일대일 대응시키는" 변환기를 만드는 것이다. 변환기는 시각 속성(폰트, 색, 들여쓰기 등) 을 스스로 결정하지 않는다. 변환기의 책임은 마크다운의 모든 구조 단위에 올바른 HWPX 스타일 이름을 부착하는 데까지이며, 시각 속성의 최종 조정은 사용자가 한/글 스타일 편집창에서 수행한다.

이 설계 선택이 프로젝트의 가장 중요한 전제이다. 변환기가 시각 속성까지 책임지려 하면 사용자 요구사항만큼 템플릿이 분기하여 통제 불가능해진다. 반대로 스타일 이름만 정확히 부착하면, 사용자는 한/글에서 스타일을 한 번 수정하여 문서 전체에 일괄 적용할 수 있다.

### 설계 철학

세 가지 원칙을 따른다.

첫째, 스타일 이름 카탈로그를 고정한다. 스타일 이름을 사전에 정의하고 header.xml 템플릿과 코드 양쪽에서 동일한 ID 순서로 참조한다. 사용자가 이름을 바꾸고 싶을 때는 외부 YAML 한 파일만 교체한다.

둘째, 마크다운의 구조를 손실 없이 스타일로 환원한다. 제목 레벨, 목록 깊이, 표 셀의 머리/본문 구분, 캡션, 각주, 참고문헌이 모두 별개의 스타일로 분리되어야 한다.

셋째, 코어 엔진은 한 사람이 통째로 재작성할 수 있을 만큼 작고 응집적이어야 한다. 2일 스프린트에서 가장 큰 위험은 핵심 로직의 난맥상이며, 이를 방지하려면 코어를 최강 개발자 한 사람이 전담하고 주변 모듈은 명확한 인터페이스로 분리해야 한다.

### 기술 스택

언어는 Python 3.11 이상을 쓴다. 이는 마크다운 파서(markdown-it-py), XML 빌더(lxml), YAML 로더(pyyaml), ZIP 패키징(내장 zipfile) 이 모두 성숙한 표준 생태계를 갖추고 있기 때문이다.

의존성은 다음으로 최소화한다. markdown-it-py 가 AST 토큰화, lxml 이 XML 빌딩과 C14N 정규화, pyyaml 이 스타일 매핑 로딩, Pillow(선택) 가 이미지 크기 읽기를 담당한다. 그 외에는 표준 라이브러리만 사용한다. 수식 변환에는 Anthropic 또는 OpenAI 의 LLM API 가 선택적으로 사용된다.

배포 형식은 pip 설치 가능한 Python 패키지이다. CLI 엔트리포인트는 `mapsi` 이며, 라이브러리로도 임포트 가능하게 설계한다.

### 전체 아키텍처

파이프라인은 선형이며 다섯 단계를 거친다.

```
┌─────────────┐   ┌─────────────┐   ┌──────────────┐   ┌─────────────┐   ┌─────────────┐
│  md 원본    │ → │  AST 파싱   │ → │ 스타일 디스패치│ → │ HWPX 빌드   │ → │ ZIP 패키징  │
│ (input.md)  │   │ markdown-it │   │ styles.yaml  │   │ lxml builder│   │ zipfile     │
└─────────────┘   └─────────────┘   └──────────────┘   └─────────────┘   └─────────────┘
                                                                                 │
                                                                                 ▼
                                                                        ┌─────────────┐
                                                                        │ output.hwpx │
                                                                        └─────────────┘
```

각 단계의 산출물은 Python 객체로 메모리에 유지되며, 마지막 ZIP 패키징 단계에서만 디스크에 쓰인다. 이미지가 포함된 경우에는 BinData 디렉토리를 임시 작업공간에 만들고 content.hpf 의 manifest 를 동적으로 갱신한다.

## 팀 구성과 역할 원칙

### 팀원 정의

편의상 본 문서 전체에서 팀원을 A, B, C 로 지칭한다.

A 는 HWPX 전문가이다. 한/글의 스타일 시스템과 HWPX 내부 XML 구조에 대한 감이 가장 뛰어나며, 생성된 결과물이 정답인지 판정한다. 바이브 코딩에 능숙하므로 후반 UI 폴리싱과 샘플 생성에 기여한다.

B 는 최고 실력의 개발자이다. 코어 엔진(파서, 빌더, 디스패처) 을 전담하며, 비상 시 프로젝트 전체 코드를 재작성할 수 있는 수준이 요구된다.

C 는 숙련된 개발자이며, 주변 인프라(패키징, I/O, CLI, 테스트) 를 담당한다. B 가 정의한 인터페이스에 따라 구현하며, 코어 로직은 건드리지 않는다.

### B 중심 설계

2일 스프린트에서 가장 치명적인 실패는 "두 개발자가 각자 XML 빌더를 만들다가 충돌하는 것" 이다. 이를 피하기 위해 코어 엔진은 B 한 사람이 통째로 소유하며, C 는 코어를 호출하는 쪽에만 관여한다.

이 구조의 장점은 네 가지이다. 첫째, 코어의 설계 철학이 일관된다. 둘째, 비상 상황에서 B 가 코어 전체를 한 번에 수정할 수 있다. 셋째, C 는 코어 내부를 몰라도 작업할 수 있어 병렬성이 높다. 넷째, B 와 C 사이의 소통 비용이 인터페이스 계약 몇 개로 축소된다.

단점은 B 의 부하가 크다는 점이다. 이를 완화하기 위해 B의 주도 하에 완결성을 해치지 않는 범위에서 타 개발자에게 적절히 작업을 배분한다. 

### 인터페이스

B 와 C 사이의 경계는 다음 다섯 개의 함수 시그니처로 완전히 정의된다. 1일차 오전 체크포인트 이전에 B 가 이 다섯 개를 확정하고 두 사람이 합의한다. 이후에는 인터페이스 변경을 허용하지 않는다.

```python
# 계약 1. 스타일 매핑 로드. C가 구현, B가 호출.
def 스타일매핑로드(yaml경로: str) -> dict: ...

# 계약 2. ZIP 패키징. C가 구현, B가 호출.
def HWPX패키징(작업디렉토리: str, 출력경로: str) -> None: ...

# 계약 3. 이미지 등록. C가 구현, B가 호출.
def 이미지등록(원본경로: str, 작업디렉토리: str) -> tuple[str, dict]: ...
# 반환값은 (binary_item_id, manifest_entry_dict)

# 계약 4. Manifest 갱신. C가 구현, B가 호출.
def manifest갱신(content_hpf경로: str, 추가항목: list[dict]) -> None: ...

# 계약 5. CLI 엔트리포인트. C가 구현.
def 메인(argv: list[str]) -> int: ...
# 내부에서 B의 변환함수(마크다운toHWPX)를 호출.
```

반대 방향의 계약, 즉 B 가 구현하고 C 가 호출하는 함수는 하나이다.

```python
# 계약 6. 마크다운 to HWPX 변환의 코어. B가 구현, C가 호출.
def 마크다운toHWPX(md경로: str,
                    출력경로: str,
                    스타일맵: dict,
                    작업디렉토리: str) -> None: ...
```

### 샘플 디렉토리의 구성 원리

레퍼런스 샘플은 변환기 템플릿과 회귀 테스트의 원천이 되는 저장소이다. 회귀 테스트는 pytest 가 각 마크다운을 변환기에 통과시켜 나온 결과를 기대 HWPX 와 XML 단위로 비교하는 방식으로 수행된다. 각 증분 단계의 샘플 문서는 증분(incremental) 형태로 구성되며, 이전 단계와 비교하면 HWPX 의 어떤 XML 요소와 스타일 정의가 해당 마크다운 요소의 추가로 인해 영향을 받는지 정확히 식별할 수 있다.

각 샘플 디렉토리는 세 가지 파일을 함께 보관한다. `NN_xxx.md` 는 변환기의 기대 입력이며, `NN_xxx.hwpx` 는 그 입력이 생성해야 할 정답 출력 단일 파일이고, `unpacked/` 는 hwpx 의 압축 해제본이다. hwpx 단일 파일은 한/글이 바로 열 수 있는 형태이고, unpacked 는 Git 이 내부 XML 변경을 추적할 수 있도록 펼쳐놓은 형태이다. md 파일은 해당 증분이 새로 도입하는 요소만 포함하는 최소 패턴이며, 상단 YAML front matter 에 도입 요소, 스타일 매핑, 테스트 관점, 경계 조건이 명시되어 있다.

`base/` 는 쪽번호, 용지, 여백, 기본 글꼴, 메모 스타일 같은 문서 전역 설정이 완비된 최소 문서로서 모든 증분 샘플의 출발점이 된다.

### B 담당 범위 상세

B 는 다음 파일들을 커버한다. 코어 엔진에 해당하며, 비상 시 재작성 대상이 되는 범위이다.

`converter.py` 는 전체 파이프라인의 오케스트레이터이다. 입력 md 경로와 출력 hwpx 경로를 받아 파서, 빌더, 패키저를 순서대로 호출한다. 이 파일의 코드 길이는 50 줄을 넘기지 않도록 유지한다.

`parser.py` 는 markdown-it-py 의 토큰 스트림을 블록 트리로 재구성한다. 출력 타입은 Block 데이터클래스의 리스트이며, Block 은 역할(문자열), 깊이(정수), 자식(리스트), 텍스트(문자열), 메타(딕셔너리) 필드를 가진다.

`ast_walker.py` 는 Block 트리를 순회하며 이벤트 시퀀스를 뱉는다. 이 분리의 이유는 표 캡션 감지(표 직전 단락의 승격), 그림 캡션 감지(그림 직후 단락의 승격), 참고문헌 섹션 감지(참고문헌 제목 이후 단락의 승격) 같은 문맥 의존 규칙을 여기에 몰아넣기 위함이다.

`styles.py` 는 역할 키워드(문자열) 와 깊이(정수) 를 받아 HWPX 스타일 ID(정수) 를 반환하는 순수 함수를 제공한다. 외부 YAML 에서 로드된 매핑 딕셔너리를 주입받으며, 테스트 가능성이 높다.

`builder/section.py` 는 Block 트리를 받아 완성된 section0.xml 문자열을 반환한다. 내부에서 `builder/elements.py` 의 헬퍼들을 조립하여 사용한다.

`builder/elements.py` 는 개별 XML 노드 빌더들을 모아둔다. 문단빌더, 런빌더, 표빌더, 그림빌더, 각주빌더 등이 여기에 있다.

`builder/equation.py` 는 hp:equation 요소를 생성하며, 내부에서 `math/converter.py` 를 호출하여 LaTeX 를 HNC 수식으로 변환한다.

`builder/header.py` 는 스타일 정의가 담긴 header.xml 을 로드해 반환한다. 대부분의 스타일 속성은 템플릿에 이미 정의되어 있으므로 로드 후 그대로 반환하는 단순한 구조이다.

### C 담당 범위 상세

C 는 다음 파일들을 담당한다. 코어를 건드리지 않으며, B 가 정의한 계약을 구현한다.

`cli.py` 는 argparse 로 커맨드라인 인자를 파싱하고 `converter.마크다운toHWPX()` 를 호출한다.

`config.py` 는 `spec/styles.yaml` 을 로드하여 스타일 매핑 딕셔너리를 반환한다.

`packager.py` 는 작업 디렉토리를 ZIP 으로 패키징한다. mimetype 을 무압축 첫 엔트리로 저장하는 것이 핵심 책임이다.

`builder/manifest.py` 는 content.hpf 의 XML 을 파싱하고 manifest 항목을 추가 삽입한다. 이미지가 등록될 때마다 호출된다.

`builder/bindata.py` 는 원본 이미지 파일을 BinData/ 로 복사하고 고유 ID 를 발급한다.

`math/converter.py` 는 LaTeX 수식을 한/글 HNC 수식 문법으로 변환한다. Anthropic 또는 OpenAI API 키가 환경 변수에 설정되어 있으면 LLM 을 호출하고, 없으면 `[hnc 수식]` 태그를 붙여 원문을 보존하는 폴백을 수행한다.

`math/cache.py` 는 변환 결과를 로컬 JSON 파일에 캐시하여 반복 변환의 비용을 줄인다.

`tests/test_packager.py`, `tests/test_integration.py` 는 C 가 작성하는 검증 테스트이다.

### A 담당 범위 상세

A 는 코드 구현에 직접 참여하지 않으며, 다음 산출물을 책임진다.

`samples/` 디렉토리의 레퍼런스 HWPX 10 개를 한/글에서 직접 작성한다. 각 샘플은 .hwpx 원본 파일과 압축 해제한 XML 디렉토리 두 형태로 커밋한다. base 샘플 1 개와 incremental 샘플 9 개로 구성된다. 더불어 각 샘플의 기대 입력이 되는 마크다운 파일(NN_xxx.md) 을 같은 폴더에 작성하여 변환기의 TDD 입력으로 제공한다. 각 md 상단 YAML front matter 에 도입 요소, 스타일 매핑, 테스트 관점, 경계 조건을 명시한다.

`spec/styles.yaml` 에 스타일 이름 카탈로그를 확정한다.

`spec/mapping.md` 에 마크다운 요소와 HWPX 스타일 간의 매핑 표를 작성한다.

`spec/math_examples.yaml` 에 LaTeX-HNC 변환의 정답 쌍 20 - 30 개를 수집하여 LLM 의 few-shot 예시로 활용한다.

`templates/` 디렉토리의 header.xml 을 한/글에서 저장한 HWPX 의 header.xml 을 기반으로 다듬는다. C 가 템플릿의 정적 파일 골격을 잡아주고, A 가 스타일 정의 부분을 검수한다.

`docs/user_guide.md` 와 한/글 스타일 편집 가이드를 작성한다.

각 단계마다 B 와 C 가 만든 HWPX 를 한/글에서 열어 스타일이 올바르게 분리되어 부착되었는지 검증한다.

## 스타일 카탈로그

코드와 템플릿 양쪽에서 참조되는 스타일 목록이다. 한/글 내장 스타일인 "개요 1 - 6" 과 "본문" 을 제외한 나머지는 사용자 스타일로 등록된다.

### 스타일 이름 목록

```
본문                    ← 한/글 내장
개요 1                  ← 한/글 내장 (마크다운 # 에 대응)
개요 2                  ← 한/글 내장 (마크다운 ## 에 대응)
개요 3                  ← 한/글 내장 (마크다운 ### 에 대응)
개요 4                  ← 한/글 내장 (마크다운 #### 에 대응)
개요 5                  ← 한/글 내장 (마크다운 ##### 에 대응)
개요 6                  ← 한/글 내장 (마크다운 ###### 에 대응)
네모                    ← 순서 없는 목록 1 단계 (■)
동그라미                ← 순서 없는 목록 2 단계 (●)
줄                      ← 순서 없는 목록 3 단계 (-)
번호1                   ← 순서 있는 목록 1 단계
번호2                   ← 순서 있는 목록 2 단계
번호3                   ← 순서 있는 목록 3 단계
인용                    ← blockquote 대응
코드                    ← code block 대응
표내용                  ← GFM 표 셀
표캡션                  ← 표 제목
그림                    ← 이미지 단락
그림캡션                ← 이미지 제목
각주                    ← 한/글 내장, 각주 본문
참고문헌                ← 매달린 들여쓰기로 구성된 서지 항목
메모                    ← 한/글 내장, 문서 메모 본문
```

한/글 스타일 파일에는 기본적으로 "개요 1" 에서 "개요 10" 까지 제시되어 있다. 마크다운의 `#` 은 개요 1, `##` 은 개요 2 와 같은 식으로 `######` 까지 6 단계가 대응된다. 나머지 개요 7 에서 개요 10 은 한/글 문서에서 6 단계 이상의 깊은 하위 절을 표현하기 위한 설정이며, 마크다운과는 대응되지 않으나 한/글에서 기본적으로 제공하는 설정이어서 변환 후 사용자가 한/글 편집 단계에서 필요에 따라 직접 적용할 수 있도록 템플릿에 보존한다. 아래에서는 직관적인 의미를 위해 개요 1 의 스타일로 표현된 내용을 "제목 1" 이라고 표시한다. 한/글의 개요는 사실상 제목의 의미이다.

### 마크다운 요소와 스타일의 대응

| 마크다운 요소 | 조건 | HWPX 스타일 |
|---|---|---|
| 평문 단락 | 어느 블록 컨테이너에도 속하지 않음 | 본문 |
| `# 제목` | 헤딩 레벨 1 | 개요 1 |
| `## 제목` | 헤딩 레벨 2 | 개요 2 |
| `### 제목` | 헤딩 레벨 3 | 개요 3 |
| `#### 제목` | 헤딩 레벨 4 | 개요 4 |
| `##### 제목` | 헤딩 레벨 5 | 개요 5 |
| `###### 제목` | 헤딩 레벨 6 | 개요 6 |
| `* 항목` | 순서없는, depth 1 | 네모 |
| `  * 항목` | 순서없는, depth 2 | 동그라미 |
| `    * 항목` | 순서없는, depth 3 | 줄 |
| `1. 항목` | 순서있는, depth 1 | 번호1 |
| `   1. 항목` | 순서있는, depth 2 | 번호2 |
| `      1. 항목` | 순서있는, depth 3 | 번호3 |
| 표 셀 (전체) | GFM table | 표내용 |
| 표 캡션 | 표 직전 단락이 "표 1." 또는 "Table 1." 로 시작 | 표캡션 |
| 이미지 | `![](url)` 의 컨테이너 문단 | 그림 |
| 그림 캡션 | 그림 직후 단락이 "그림 1." 또는 "Figure 1." 로 시작 | 그림캡션 |
| 코드 블록 | 백틱 세 개로 감싼 블록 | 코드 |
| 인용 | `>` 로 시작하는 블록 | 인용 |
| 각주 본문 | `[^N]` 정의 | 각주 |
| 참고문헌 | "# 참고문헌" 또는 "# References" 섹션 하위 단락 | 참고문헌 |
| 수식 (인라인/블록) | `$...$` 또는 `$$...$$` | hp:equation (스타일 없음, LaTeX → HNC 변환) |

### 번호 처리 규칙

위 매핑에서 번호와 관련된 두 경로는 방향이 반대이며, 혼동을 피하기 위해 명시적으로 구분한다.

순서 있는 목록(번호1/2/3) 은 원문 번호 텍스트를 그대로 보존한다. 마크다운의 "1. ", "2. " 등은 hp:t 의 본문 텍스트로 삽입되며 한/글의 자동 번호매김은 사용하지 않는다. md 원문이 "1. / 1. / 1." 이면 세 항목 모두 "1." 로 출력되고, "1. / 2. / 3." 이면 그대로 출력된다. 근거는 번호1/2/3 스타일이 깊이별 들여쓰기만 제공할 뿐 자동 번호 메커니즘을 가지지 않기 때문이다.

표 캡션과 그림 캡션은 반대로 접두사를 제거하고 한/글 autoNum 에 번호 부여를 위임한다. 변환기가 정규식 "^(표|Table|그림|Figure)\s+\d+\.\s*" 로 감지하면 접두사를 파싱 후 제거하고 뒤쪽 본문만 캡션 슬롯에 저장한다. 예를 들어 md 의 "표 1. 주요 재정 융자 사업 비교" 는 캡션 슬롯에 "주요 재정 융자 사업 비교" 로만 기록되며, 번호 "1" 은 한/글이 문서 내 표 등장 순서대로 자동 부여한다. 사용자가 md 에서 다른 번호("표 5." 등) 를 써도 무시된다. 근거는 hwpx 의 표와 그림이 autoNum 메커니즘을 기본 제공하므로 문서 편집 중 표/그림 추가 삭제 시 자동 재조정되는 실용성이 크기 때문이다.

## 2일 작업 플로

총 가용 시간은 3 인 × 16 시간 = 48 인시(man-hour) 이다. 하루를 오전(4 시간) 과 오후(4 시간) 로 나누어 네 개의 체크포인트를 둔다.

### 1 일차 오전 (0 - 4 시간) - 초기화와 계약 확정

A 는 본 프로젝트에서 이미 10 개의 레퍼런스 HWPX 샘플을 사전 준비하였다. 이 샘플들을 기반으로 Claude Code 를 실행하여 template 자동 추출을 수행한다. 이후 `spec/styles.yaml` 과 `spec/mapping.md` 를 최종 확정한다.

B 는 A 가 제공한 base 샘플을 압축 해제하여 `templates/` 디렉토리의 정적 파일 구조를 파악한다. 병행하여 인터페이스 계약 6 개를 확정해 `spec/interfaces.md` 에 기록한다. 프로젝트 스캐폴드(pyproject.toml, 패키지 디렉토리 구조) 를 만들고 `converter.py`, `parser.py`, `builder/section.py` 의 뼈대(함수 시그니처와 docstring 만 있는 상태) 를 커밋한다.

C 는 B 가 인터페이스 계약을 커밋하면 `packager.py`, `config.py`, `cli.py`, `builder/manifest.py`, `builder/bindata.py` 의 스켈레톤을 만든다. 이 시간대 가장 중요한 작업은 "빈 본문으로 HWPX 를 만들어 한/글에서 열리는 것을 확인" 하는 패키징 스모크 테스트를 완성하는 것이다.

이 구간의 결과물: 리포지토리, 인터페이스 계약 문서, 빈 HWPX 가 한/글에서 정상 오픈되는 것 확인, Claude Code 를 통한 template 자동 추출 완료, styles.yaml 확정.

### 1 일차 오후 (4 - 8 시간) - 블록 스타일 일차 구현

B 는 블록 수준 스타일 매핑을 순차적으로 구현한다. 이 시점에서 인라인 서식(굵게, 기울임, 인라인 코드 등) 은 다루지 않는다. 구현 순서는 평문 단락, 제목 6 레벨, 순서없는 목록 depth 1 - 3, 순서있는 목록 depth 1 - 3, 인용, 코드블록 순이다.

C 는 테스트 하네스를 완성한다. `tests/fixtures/` 에 마크다운 입력 파일 10 개 정도를 두고, 각각 기대하는 section0.xml 을 A 가 제공한 샘플에서 추출해 저장한다. 비교 도구는 lxml 의 C14N 정규화를 거친 후 의미 있는 차이만 보여주는 방식으로 만든다.

A 는 B 와 C 가 생성하는 HWPX 파일을 한/글에서 열어 각 문단이 기대한 스타일로 귀속되는지 스타일 창에서 직접 확인한다. 문제가 있으면 정답 XML 조각을 잘라 B 에게 전달한다.

이 구간의 결과물: 제목 1 - 6, 평문 단락, 순서없는 목록 3 레벨까지, 순서있는 목록 3 레벨까지, 인용, 코드블록이 한/글에서 올바른 스타일로 렌더링됨을 확인.

### 2 일차 오전 (8 - 12 시간) - 복잡 요소 구현

B 는 남은 블록 요소를 구현한다. 우선순위는 표(표내용, 표캡션), 그림(그림, 그림캡션), 각주, 참고문헌 순이다. 표의 HWPUNIT 계산은 페이지 폭 균등 분할로 단순화한다. 이미지는 C 가 만든 `이미지등록()` 함수를 호출해 BinData 복사와 manifest 갱신을 위임한다.

C 는 `이미지등록()`, `manifest갱신()`, 이미지 파이프라인 전체를 완성한다. 원본 파일 누락, 지원 안 되는 포맷(.svg 등), 동일 이미지 중복 등록 같은 예외 상황을 처리한다. 병행하여 수식 변환 모듈(`math/converter.py`, `math/cache.py`) 을 구현한다. 환경 변수 검사, API 호출 분기, 캐시 조회/저장을 포함한다.

A 는 B 가 생성한 각 복잡 요소의 HWPX 를 한/글에서 열어 검증한다. 특히 각주 번호 자동 매김과 참고문헌 매달린 들여쓰기가 스타일 편집으로 조정 가능한지 확인한다. 수식 변환 검증을 위해 `spec/math_examples.yaml` 에 LaTeX-HNC 정답 쌍을 수집한다.

이 구간의 결과물: 주요 스타일의 실제 마크다운 입력에서의 정상 생성 확인.

### 2 일차 오후 (12 - 16 시간) - 인라인 서식과 마감

B 는 인라인 서식을 구현한다. 굵게, 기울임, 취소선, 인라인 코드, 하이퍼링크를 charPr 매핑으로 처리한다. 블록 레벨이 안정된 뒤에 붙이는 이유는 디버깅 경로를 분리하기 위함이다. 수식 빌더(`builder/equation.py`) 도 이 시간대에 완성한다.

C 는 실제 학술 문서 형태의 통합 테스트 2 - 3 개를 돌리고 실패하는 항목을 B 에게 보고한다. CLI 옵션을 정비한다(`--style-map` 경로 지정, `--verbose` 로그, `--no-llm` 수식 변환 비활성화, `--dry-run` 모드 등). `pyproject.toml` 을 정리해 `pip install -e .` 로 설치 가능하게 한다.

A 는 자신의 실제 마크다운 원고 하나를 입력으로 넣어 나온 HWPX 를 한/글에서 열고, 스타일 편집창에서 각 스타일을 원하는 모양으로 바꿔 본 뒤, 일괄 반영되는지 확인한다. README 와 스타일 편집 가이드를 마무리한다.

이 구간의 결과물: 0.1.0 릴리스 태깅 가능한 상태.

## 체크포인트별 합격 기준

각 체크포인트는 게이트이다. 통과하지 못하면 전원이 해당 시점의 문제 해결에 투입된다.

### 체크포인트 1 (4 시간 경과)

빈 마크다운(제목 1 개와 단락 1 개 정도) 을 변환한 HWPX 가 한/글에서 오류 없이 열려야 한다. 스타일은 잘못되어도 되지만 파일 자체는 열려야 한다. 이것이 실패하면 ZIP 패키징 또는 XML 네임스페이스 문제이며, 이후 모든 작업의 전제가 무너진다.

### 체크포인트 2 (8 시간 경과)

다음 마크다운이 올바른 스타일로 변환되어야 한다.

````markdown
# 제목 1
## 제목 2

평문 단락이다.

* 1 단계 항목
  * 2 단계 항목
    * 3 단계 항목

1. 순서 1
   1. 순서 1.1

> 인용문

```
코드 블록
```
````

한/글에서 열어 스타일 창에서 각 문단의 스타일 이름을 확인하는 방식으로 검증한다.

### 체크포인트 3 (12 시간 경과)

표(캡션 포함), 그림(캡션 포함), 각주 하나, 참고문헌 섹션 3 개 항목, 수식 2 개가 포함된 마크다운이 변환되어 각각의 요소가 올바른 스타일을 가진다.

### 체크포인트 4 (16 시간 경과)

실제 학술 문서 분량(5000 자 이상, 표 2 개, 그림 2 개, 각주 5 개, 참고문헌 10 개 이상, 수식 포함) 의 마크다운이 변환되어 한/글에서 정상 편집 가능하다. 사용자가 한/글 스타일 편집창에서 한 스타일(예, 개요 1) 의 속성을 바꾸면 해당 스타일을 가진 모든 문단이 일괄 변경된다.

## 수식 처리

마크다운의 수식 문법(`$...$` 인라인과 `$$...$$` 디스플레이) 에 대응하는 hwpx 요소를 생성한다. hwpx 는 수식을 hp:equation 요소로 표현하며 내부 hp:script 에 한/글 자체의 수식 언어(HNC 수식) 로 작성된 문자열을 담는다. HNC 수식은 LaTeX 와 문법이 유사하지만 세부 명령어와 구조가 다르므로 변환기가 LaTeX 를 HNC 수식으로 변환해야 한다.

결정론적 규칙 엔진으로 완전히 구현하려면 LaTeX 전체 문법을 파싱해야 하므로 범위가 커 2 일 스프린트 내에 구현하기 어렵다. 대신 변환기는 LLM API 의 사용 가능 여부에 따라 두 경로를 선택한다. 환경 변수에 `OPENAI_API_KEY` 또는 `ANTHROPIC_API_KEY` 가 설정되어 있으면 LLM 을 호출하여 LaTeX 를 HNC 수식으로 변환하고, 그 결과를 hp:script 에 삽입한다. API 키가 없으면 LaTeX 원문을 그대로 보존하되 앞에 `[hnc 수식]` 태그를 붙여 사용자가 변환 결과에서 해당 위치를 찾아 한/글 수식 편집기로 직접 입력할 수 있도록 안내한다.

LLM 호출 시 한/글이 공개한 수식 스펙 문서(`spec/hnc_equation_spec.pdf`) 를 컨텍스트로 함께 전달하여 변환 정확도를 높인다. 반복 호출을 줄이기 위해 변환 결과를 로컬 JSON 캐시(`~/.mapsi/equation_cache.json`) 에 (LaTeX 원문 → HNC 수식) 쌍으로 저장하며, 같은 LaTeX 가 다시 등장하면 캐시에서 바로 반환한다.

## 비상 계획

### 시나리오 1. C 가 막힘

C 가 패키징 또는 이미지 파이프라인에서 막힐 경우, B 가 해당 모듈을 인수한다. 인터페이스는 이미 확정되어 있으므로 구현체만 바꾸면 되며, 기존 C 의 테스트가 회귀 방지 역할을 한다.

### 시나리오 2. B 가 막힘

B 가 HWPX XML 구조 문제에서 막힐 경우, A 가 해당 요소의 정답 XML 을 직접 제공하고 B 는 그 XML 을 생성하도록 빌더를 역공학한다. 공식 스펙 문서를 뒤지는 것보다 A 의 샘플이 훨씬 빠른 경로이다.

### 시나리오 3. 코어 설계 오류

B 가 1 일차 오후 중반에 설계 오류를 발견하면, 즉시 C 와 A 에게 알리고 코어 재설계에 들어간다. 재설계 시간은 최대 2 시간으로 제한하며, 그 안에 끝나지 않으면 스코프를 축소한다(인라인 서식, 각주, 참고문헌 등을 0.1 에서 제외).

### 시나리오 4. 한/글이 생성 파일을 열지 못함

mimetype 의 무압축 저장 누락, XML 선언 누락, 네임스페이스 불일치 중 하나가 원인일 확률이 95% 이상이다. C 가 `packager.py` 의 세 가지 항목을 체크리스트로 검증한다.

### 시나리오 5. LLM 수식 변환 실패

LLM API 호출이 실패하거나 결과가 부정확하면 즉시 폴백 태그(`[hnc 수식]` + LaTeX 원문) 로 대체한다. 사용자는 한/글에서 해당 위치를 찾아 수식을 직접 입력한다. 이 경우 LLM 의존 없는 규칙 기반 폴백 구현은 0.2 로 미룬다.

### 시나리오 6. 스코프 축소의 우선순위

시간이 부족할 경우 다음 순서로 기능을 0.1 에서 제외한다. 수식 LLM 변환(폴백만 유지), 각주, 참고문헌, 인라인 서식의 일부(취소선, 하이퍼링크). 단, 목록 3 레벨과 제목 6 레벨은 절대 줄이지 않는다. 스타일 분리의 핵심 가치가 여기에 있기 때문이다.

## 최종 산출물 체크리스트

2 일 스프린트 종료 시점에 다음 항목이 리포지토리에 존재해야 한다.

- mapsi 파이썬 패키지가 `pip install -e .` 로 설치 가능
- `mapsi input.md -o output.hwpx` 커맨드가 동작
- `templates/` 에 header.xml 을 포함한 정적 파일 세트가 완비
- `spec/styles.yaml`, `spec/mapping.md`, `spec/interfaces.md`, `spec/math_examples.yaml` 이 최신 상태
- `samples/` 에 10 개 레퍼런스 HWPX 파일과 unpacked 디렉토리가 있음
- `spec/references/` 에 한/글 공식 문서 5 종이 보존되어 있음
- `tests/` 에 fixtures 10 개 이상과 통합 테스트 2 - 3 개가 있으며, pytest 로 전부 통과
- README.md 와 한/글 스타일 편집 가이드가 완성
- 실제 학술 문서 분량의 마크다운을 변환한 HWPX 가 한/글에서 편집 가능
- 스타일 편집으로 문서 전체 시각 속성의 일괄 변경이 가능
