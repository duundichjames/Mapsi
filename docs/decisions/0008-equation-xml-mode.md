# ADR 0008 — 수식 hp:equation XML 모드 채택

상태: Accepted (2026-05-30, `feature/equation-xml-mode` 시점)
관련: ADR 0002 (대체 대상), `mapsi/math/latex_parser.py`, `mapsi/math/hnc.py`,
`mapsi/builder/equation.py`, `mapsi/builder/elements.py`, `mapsi/parser.py`,
`spec/equation_samples/한글수식표본.hwpx`

## 배경

ADR 0002 는 수식을 평문 마커 모드로 변환하기로 결정하였다. 마크다운의 인라인
수식과 디스플레이 수식을 한/글 본문 `hp:t` 안에 `[hnc 수식] ... [/hnc 수식]`
형태의 평문으로 박아 두고, 사용자가 한/글에서 그 마커를 찾아 수식 편집기에
직접 입력하거나 복사해 붙여 넣도록 하는 방식이었다. 이 방식은 한/글
`hp:equation` 스키마의 호환 리스크와 구현 작업량을 회피하기 위한 현실적
타협이었으며, 당시에는 정답 표본조차 수식을 평문으로 회피하고 있었으므로
합리적 선택이었다.

그러나 평문 마커 방식은 사용자 개입을 요구한다는 본질적 한계를 안고 있었다.
변환된 문서를 한/글에서 열면 수식이 평문 마커로 보이며, 사용자가 마커를
일일이 찾아 수식 편집기에 수동으로 재입력하거나 마커 안 문자열을 복사해
붙여 넣어야 비로소 렌더링되었다. 수식이 많은 학술 원고에서 이 수동 단계는
변환 자동화의 가치를 크게 깎아내렸다.

v0.2 의 목표는 이 수동 단계를 제거하는 것이다. 사용자가 변환된 문서를
한/글에서 열기만 하면 별도 조작 없이 즉시 렌더링되는 수식을 생성하여야 한다.
이를 위해서는 평문 마커가 아니라 한/글이 수식으로 인식하는 진짜
`hp:equation` 객체를 정확한 구조로 생성해야 한다.

## 결정

평문 마커 대신 `hp:equation` XML 노드를 직접 생성하는 방식으로 전환한다.
LaTeX 수식을 한/글 HNC 수식 문법으로 규칙 기반 결정론적으로 변환하고, 그
결과 문자열을 `hp:equation` 의 `hp:script` 슬롯에 담는다. 변환은 외부 LLM 에
의존하지 않으며, 오프라인 환경과 CI 에서도 동일한 결과를 낸다.

생성하는 `hp:equation` 의 고정 속성과 자식 구조는 한/글이 직접 만든 수식
표본 `spec/equation_samples/한글수식표본.hwpx` 를 분석하여 확정하였다. 한/글이
수식을 다시 열 때 재계산하는 크기 같은 값은 표본의 흔한 기본값을 사용한다.

## 구현 구조

구현은 단계 1 - 4 로 구성된다.

LaTeX 파서가 첫 단계를 맡는다. 토크나이저가 LaTeX 문자열을 명령어, 중괄호,
첨자, 정렬 기호, 행 구분, 일반 문자로 끊고, 재귀 하향 파서가 이를 구문
트리로 세운다. 행렬과 정렬과 분기 같은 환경은 행과 칸의 2 차원 구조를 가진
전용 노드로 묶는다. 파싱이 불가능한 입력은 예외를 던지지 않고 원본을 보존한
실패 결과로 돌려준다.

HNC 변환이 둘째 단계를 맡는다. LaTeX 명령어를 HNC 토큰으로 대응시키는
매핑은 코드가 아니라 데이터 테이블로 분리하여 분수와 근호, 큰 연산자,
그리스 문자, 관계 기호, 화살표, 장식, 글꼴, 구분기호, 함수명을 항목 추가만으로
확장할 수 있게 하였다. 첨자 사이 공백과 항 구분 공백 같은 표기 규칙은 한/글
표본의 경향을 따른다.

`build_equation` 빌더가 셋째 단계를 맡아 변환 결과를 `hp:equation` 노드로
조립한다. `elements.py` 가 마지막 단계로 이 빌더를 호출하여 본문 `hp:run`
안에 수식을 끼워 넣는다.

입력 인식은 세 가지 구문을 각각의 markdown-it 플러그인으로 처리한다. 달러
구문 `$ ... $` 와 `$$ ... $$` 는 dollarmath 가, 환경 구문 `\begin{...} ...
\end{...}` 는 amsmath 가, 백슬래시 구문 `\( ... \)` 와 `\[ ... \]` 는 texmath
의 brackets 프리셋이 맡는다. 세 플러그인은 구분자가 겹치지 않아 충돌 없이
공존하며, 모두 토큰 기반이라 코드 스팬과 코드 펜스 안의 수식 구문은 건드리지
않는다.

## 핵심 설계 결정

`hp:equation` 은 `hp:ctrl` 로 감싸지 않고 `hp:run` 의 직접 자식으로 둔다.
이는 한/글 수식 표본을 분석한 실측 결과를 따른 것으로, 인라인 수식이 든
`hp:run` 의 자식 순서는 평문, 수식, 평문 형태가 된다.

인라인과 디스플레이의 구분은 `hp:equation` 자체 속성이 아니라 단락 배치로
표현한다. 표본 분석에서 두 경우의 `hp:equation` 구조가 동일하고 차이는 단락
배치에서만 드러났으므로, 빌더는 양쪽에 같은 노드를 생성하고 디스플레이
여부는 상위 단락 조립 단계에서 처리한다.

변환할 수 없는 토큰을 만나면 폴백으로 LaTeX 원문을 `hp:script` 에 보존한다.
매핑 테이블에 없는 명령어가 하나라도 있으면 그 수식 전체를 실패로 표시하고
원문을 그대로 담아, 사용자가 한/글에서 원문을 보고 직접 보정할 수 있게 한다.
미지원 명령어는 경고 로그로 수집하여 어떤 토큰이 보완 대상인지 드러낸다.

`hp:equation` 의 `id` 는 표와 그림과 각주 같은 기존 객체와 동일하게 난수
방식으로 채번한다. 수식은 같은 객체 식별자 공간을 공유하므로 기존 관례를
따르는 것이 일관되고 충돌 확률도 균일해진다.

정렬과 크기 같은 시각 속성은 강제하지 않고 사용자가 한/글에서 조정하도록
맡긴다. 디스플레이 수식에 가운데 정렬을 강제하지 않으며 일반 본문 단락을
그대로 쓴다. 이는 구조 변환에 집중하고 시각 표현은 사용자에게 위임한다는
Mapsi 의 철학을 따른 것이다.

## 검토한 대안

변환 방식에서 매핑 테이블 방식과 파서 방식을 비교하였다. 매핑 테이블 방식은
LaTeX 토큰을 HNC 토큰으로 표면적으로 치환하는 단순한 접근이지만, 분수의
중첩이나 환경의 행과 칸 구조처럼 문맥에 의존하는 변환을 다루지 못한다. 파서
방식은 토큰을 구문 트리로 세운 뒤 트리를 순회하며 변환하므로 중첩과 환경
구조를 자연스럽게 처리하고, 새 명령어를 매핑 테이블에 더하기만 하면 확장된다.
구조적 정확성과 확장성을 위해 파서 방식을 택하였다.

폴백 단위에서 부분 변환과 수식 단위 폴백과 평문 마커 폴백을 비교하였다.
부분 변환은 변환된 부분과 실패한 부분이 한 수식 안에 섞여, 사용자가 어디까지
믿을 수 있는지 판단해야 하는 부담을 남긴다. 평문 마커 폴백은 ADR 0002 가
이미 폐기한 방식으로, 사용자 수동 개입이라는 원래 문제로 되돌아간다. 수식
단위 폴백은 실패한 수식 하나만 통째로 LaTeX 원문으로 보존하므로 경계가
명확하고, 성공한 수식은 온전히 렌더링된다. 이에 수식 단위 폴백을 택하였다.

LLM 변환은 배제하였다. LLM 은 호출 비용이 들고 외부 통신에 의존하여 오프라인
환경과 CI 에서 동작하지 않으며, 응답 속도가 변환 파이프라인의 병목이 되고,
같은 입력에 대해 비결정적 출력을 낼 수 있어 골든 회귀로 검증하기 어렵다.
코어 변환은 LLM 무관하게 항상 동일하게 동작해야 한다는 원칙에 따라 규칙 기반
결정론적 변환만 사용한다.

## ADR 0002 와의 관계

본 ADR 은 ADR 0002 의 평문 마커 모드 결정을 대체한다. 수식 변환의 활성
경로는 더 이상 평문 마커를 emit 하지 않고 `hp:equation` 노드를 생성한다.

`mapsi/math/converter.py` 와 `mapsi/math/cache.py` 는 삭제하지 않고 보존한다.
다만 `elements.py` 의 활성 경로에서는 이들의 `convert_equation` 을 더 이상
호출하지 않으며, 수식 변환은 `latex_parser` 와 `hnc` 와 `build_equation` 을
거친다. 보존한 두 모듈은 향후 다른 용도로 재사용될 여지를 남겨 둔 것이다.
