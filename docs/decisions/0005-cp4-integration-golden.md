# ADR 0005 — Phase 11 CP4 통합 골든 픽스처 도입

상태: Accepted (2026-04-19, `feature/core-engine` 시점)
관련: `docs/project_plan.md` §체크포인트 4, ADR 0001 ~ 0004,
`tests/golden/cp4_full/`, `samples/incremental/cp4_full/`

## 배경

Phase 1 ~ 10 의 누적으로 Mapsi 는 마크다운의 모든 구조 단위 — 헤딩,
번호/불릿 리스트, 인용/코드 블록, 표, 그림, 각주, 참고문헌, 수식,
인라인 서식 — 를 처리한다. 각 phase 는 독립된 단위 골든 픽스처
(`tests/golden/01_headings` ~ `tests/golden/10_inline`) 를 가지며,
이 골든들은 해당 phase 가 다루는 요소만 등장하는 10 ~ 30 줄 분량의
입력으로 정확성을 검증한다.

이 단위 골든들은 다음 두 가지 회귀를 잡지 못한다.

1. **요소가 섞일 때만 터지는 회귀.** 예) "표 바로 다음 단락이 그림
   캡션이면 들여쓰기가 깨진다", "각주 안에 인라인 코드가 있으면
   charPrIDRef 가 0 으로 떨어진다" 같은 종류. 각 요소는 단위 픽스처에서
   문제없이 통과하므로 단위 골든만으로는 잡을 수 없다.
2. **CP4 합격 기준의 기계 검증 부재.** `docs/project_plan.md`
   §체크포인트 4 는 v0.1 마일스톤의 마지막 게이트로 "실제 학술 문서
   분량 (5000 자 이상, 표 2 개, 그림 2 개, 각주 5 개, 참고문헌 10 개
   이상, 수식 포함) 의 마크다운이 변환되어 한/글에서 정상 편집 가능하다"
   를 요구한다. 그러나 이 합격 기준은 어떤 자동 테스트로도 표현되어
   있지 않아 사람이 매번 수동 확인해야 했다.

## 결정

`tests/golden/cp4_full/` 디렉토리에 **CP4 합격 기준 그대로의 학술 문서
한 편 분량 픽스처** 를 추가한다. 단위 골든의 자동 발견 메커니즘
(`tests/test_golden.py::_discover_fixtures`) 에 의해 추가 코드 변경
없이 자동으로 회귀 시나리오에 합류한다.

### 픽스처 구성

| 항목 | 값 |
|---|---|
| 본문 길이 | 5965 자 (공백 포함) |
| 헤딩 깊이 | 1 ~ 3 |
| 본문 단락 | ~30 개 |
| 번호 리스트 | 5 항목 |
| 불릿 리스트 | 2 항목 (네모 깊이 1) |
| 표 | 2 개 (캡션 포함) |
| 그림 | 2 개 (`tests/fixtures/sample_figure.png` 한 장 두 번 재활용) |
| 각주 | 5 개 |
| 참고문헌 | 11 개 (개요 1 "참고문헌" 헤딩 이후 demote) |
| 수식 | 인라인 1 개 + 디스플레이 1 개 |
| 인라인 서식 | 5 종 (bold, italic, strikethrough, code, link) 모두 |

### ADR 0004 정책 준수

ADR 0004 §결정 4 — "한 단락에 각주/수식/인라인 서식 셋 중 둘 이상
동시 사용은 v0.1 에서 `NotImplementedError` 로 거부, v0.2 에서 통합" —
를 본 픽스처도 따른다. 따라서 각주 5 개와 인라인 수식 1 개는 모두
인라인 서식 (bold/italic/strike/code/link) 이 없는 별개 단락에 배치한다.

이 제약은 본 픽스처를 작성하는 동안 한 차례 발견되어 즉시 수정됐다.
초안에서 다섯 군데가 "본문 + 인라인 서식 + 각주" 또는 "본문 + 이탤릭
+ 인라인 수식" 패턴으로 등장했고, 각각 두 개의 인접 단락으로 쪼개어
ADR 0004 정책을 만족시켰다. 이 발견 자체가 본 통합 골든의 가치를
바로 입증했다 — 단위 골든만으로는 절대 드러나지 않는 정책 위반을
한 번에 잡았다.

### 데모 페어

`samples/incremental/cp4_full/` 에 동일 마크다운 (front matter 추가 +
이미지 경로 조정) 과 변환된 `.hwpx` 를 함께 둔다. 회귀 테스트는
`tests/golden/cp4_full/` 만 사용하고, `samples/incremental/cp4_full/`
는 사람이 한컴 오피스에서 직접 열어 시각 검수하는 용도다.

## 영향

### 회귀 테스트 추가

테스트 카운트 333 → 334. 신규 1 개는 `tests/test_golden.py
::test_golden_fixture[cp4_full]`.

### v0.2 작업 시 안전망

향후 ADR 0004 §결정 4 의 통합 (인라인 + 각주/수식), 정식 hyperlink
field 도입, `hp:equation` XML 정식 발급 등의 v0.2 작업이 들어갈 때,
본 픽스처가 학술문서 한 편 단위로 회귀를 검출한다.

### 픽스처 갱신 절차

본 픽스처는 단위 골든과 달리 텍스트 시퀀스가 109 항목으로 길다.
빌더 / 파서 / 워커의 의도된 변경으로 시퀀스가 바뀌어야 할 때는
다음 절차를 따른다.

1. `python -m mapsi.cli tests/golden/cp4_full/input.md -o /tmp/cp4.hwpx --no-llm`
2. `python -m mapsi.inspect /tmp/cp4.hwpx` 로 새 시퀀스 확인
3. 변경이 의도한 것인지 사람이 한 번 검토
4. `tests/golden/cp4_full/expected.yaml` 의 `style_sequence` /
   `text_sequence` 만 갱신 (description 은 보존)

`expected.yaml` 의 `description` 필드는 본 ADR 의 의도를 픽스처에
함께 담아 두는 자료이므로, 자동 재생성 스크립트는 이 필드를
덮어쓰지 않는다.

## 거부된 대안

### 대안 1 — `tests/test_integration.py` 의 코드 기반 시나리오

`docs/c_handoff.md` §3.3 이 원래 제안했던 형태다. 거부 이유:

- **표현력 부족.** 109 단락의 시퀀스를 파이썬 assertion 으로 적으면
  가독성이 곧 0 이 된다.
- **재생성 비용.** 빌더 변경 시 새 시퀀스를 손으로 작성해야 한다.
  현재 방식은 `inspect` 의 출력을 그대로 yaml 로 떠 넣으면 끝.
- **단위 골든과의 일관성.** 단위 골든이 이미 yaml 자료 기반이라
  같은 패턴을 따르는 것이 합리적.

### 대안 2 — 한컴 오피스에서 사람이 만든 정답 .hwpx 와 바이너리 비교

거부 이유:

- 한컴이 생성하는 XML 의 의미적 동치가 보장되지 않는다 (속성 순서,
  네임스페이스 prefix, 빈 요소 표기 등).
- 변환기가 합법적으로 다른 형식의 XML 을 생성해도 회귀로 잡힌다.
- ZIP 의 압축 옵션이 다르면 바이너리 자체가 다르다.

`mapsi.inspect` 가 추출하는 (스타일 이름, 텍스트) 시퀀스는 이런
표현 차이를 정규화한 뒤의 의미적 동치 검사이므로 회귀 안전망으로
적합하다.

### 대안 3 — 사람용 데모만 두고 회귀 테스트는 추가하지 않음

거부 이유: CP4 게이트가 v0.1 마일스톤의 마감 조건이라 한 번이라도
사람이 빠뜨리면 게이트 통과 여부를 모르게 된다. 자동화는 비용이
30 분이고 효과는 영구적이다.
