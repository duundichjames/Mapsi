# ADR 0001 — 표 캡션 승격 정책 채택

- 상태: 승인 (2026-04-19)
- 작성자: Team B
- 근거 명세: `samples/incremental/05_table/05_table.md` (Team A 작성) front matter

## 배경

CommonMark / GFM 에는 "표 캡션" 을 위한 표준 문법이 없다. 즉 사용자는
표 직전에 자유 텍스트로 "표 1. 분기별 매출" 같은 줄을 적고 시각적으로만
캡션처럼 다룬다.

한/글 (HWPX) 은 표 노드(`<hp:tbl>`) 안에 명시적인 캡션 노드
(`<hp:caption>`) 를 가지며, 거기 들어간 단락은 "표캡션" 스타일을 받고
번호는 `<hp:autoNum num="N" numType="TABLE">` 으로 자동 매겨진다.

## 결정

마크다운 → 한/글 변환 시 **표 직전 단락이 정규식
`^(표|Table)\s+\d+\.\s*` 로 시작하면 표의 캡션으로 승격**한다.

- 접두사 (`표 1. `, `Table 3. ` 등) 는 파싱 후 제거하고, 그 뒤의 텍스트만
  캡션 본문으로 저장한다.
- 사용자가 적은 번호 값은 **무시**한다. 한/글이 문서 내 표 등장 순서대로
  1, 2, 3 … 을 자동 부여한다 (`<hp:autoNum>`).
- 직전 단락이 일반 단락(`role="paragraph"`) 일 때만 승격 대상. 헤딩/인용/
  코드 등은 승격하지 않는다.
- "직전" 의 정의: walker 가 평탄 Block 리스트를 순회할 때 표 Block 의
  바로 앞 인덱스. 마크다운 원본에서 빈 줄이 사이에 있어도 markdown-it 의
  토큰화 결과에서는 두 단락 사이에 빈 Block 이 들어가지 않으므로 인접
  여부는 인덱스로 판정.

## 대안과 기각 사유

### 대안 1 — 캡션 무시 (휴리스틱 미적용)
- 장점: 본문이 캡션으로 오인될 위험 0. 우리 변환기 철학("구조만 옮긴다")
  와 정합.
- 단점: 사용자가 한/글에서 매번 "이 단락을 표캡션 스타일로 바꾸기" 를
  수동으로 해야 한다. 흔한 패턴(`표 N. 제목`) 에 대한 자동화 가치를 잃음.

### 대안 2 — 명시적 마커 도입 (예: HTML 주석 `<!-- caption: ... -->`)
- 장점: 오인 위험 0. 의도가 명확.
- 단점: 마크다운 사용자가 우리 전용 문법을 학습해야 함. 다른 마크다운
  뷰어에서 캡션이 안 보임 (호환성 손실).

### 대안 3 — 채택 (A 명세 기반 정규식 승격)
- 장점: 한국어 학술 / 보고서 글쓰기에서 표준적인 캡션 표기 (`표 N.`,
  `Table N.`) 를 자동 인식. 마크다운 원본에서도 캡션처럼 읽힘
  (이중 가독성).
- 단점: 본문에서 "표 1. 의 결과를 보면 …" 같은 문장이 표 직전에 오면
  캡션으로 오인 가능. 다만 정규식 끝의 `\.\s*` 가 점 뒤 "공백 또는 끝"
  을 요구하므로 어느 정도 방어. 그래도 잠재 위험은 0 이 아님.

## 보호 장치

1. **승격 조건이 좁음**: 직전 Block 이 `role="paragraph"` 이고, 텍스트가
   정규식에 매치되며, 표가 같은 walker 순회 단계에 인접해 있어야만 발동.
2. **골든 회귀 픽스처**: `tests/golden/05_table_with_caption/` 가 있어
   캡션 동작이 의도치 않게 바뀌면 즉시 테스트가 깨진다.
3. **승격 결과 검증**: `mapsi.inspect` 로 변환 결과의 단락 시퀀스를 확인
   가능. 첫 행이 "표캡션" 스타일이면 승격됨, "본문" 으로 남으면 승격 안 됨.
4. **번호 보존 안 함의 명시**: 사용자가 적은 번호는 무시되고 한/글의
   `autoNum` 이 재부여한다. 이 차이는 `docs/c_handoff.md` 와 본 ADR 에
   기록되어 있다.

## 참고

- A 의 원 명세: `samples/incremental/05_table/05_table.md` 의 front matter
  "캡션 승격 정책" 절.
- 순서 목록의 정책과 비교: 순서 목록(03)은 원문 번호 텍스트를 *보존*
  하고 자동번호를 쓰지 않는다. 표 캡션은 반대로 원문 번호를 *제거*
  하고 한/글 autoNum 에 위임한다. 차이의 근거는 한/글의 표 자동번호
  메커니즘 존재 여부.

## 후속 — 그림 캡션으로의 일반화 (2026-04-19, Phase 6a/6b)

같은 정책을 그림에도 일반화 적용했다. 표는 *직전* 단락을 흡수하지만,
그림은 *직후* 단락을 흡수한다는 비대칭만 다르다 (학술 글쓰기 관습).

| 요소 | 정규식 | 흡수 방향 | 캡션 스타일 | autoNum 타입 |
|---|---|---|---|---|
| 표 | `^(표\|Table)\s+\d+\.\s*` | 직전 | `표캡션` | `TABLE` |
| 그림 | `^(그림\|Figure)\s+\d+\.\s*` | 직후 | `그림캡션` | `PICTURE` |

Phase 6b 부터는 그림 캡션이 별도 `hp:p` 가 아니라 `hp:pic > hp:caption`
내부에 흡수된다 (한/글 표준 모델). 흡수 직전까지의 walker 동작은 동일
하므로 ADR 본문의 보호 장치 1·2·4 는 그림에도 그대로 유효하다. 보호
장치 3 만 약간 다른데, `mapsi.inspect` 가 `hp:pic` 안의 `hp:caption`
까지 들여다보고 단락 시퀀스에 포함시키므로 동일한 검증 흐름이 가능하다.

## 후속 — 각주 본문 흡수 (2026-04-19, Phase 7)

캡션 승격과 *모양은 다르지만 동일한 정신* 의 흡수 정책을 각주에도 적용했다.
마크다운의 Pandoc 확장 각주 (`[^id]: 본문`) 정의는 별도 paragraph 처럼
파싱되지만, 한/글에서는 본문 안의 `<hp:footNote>` 노드에 임베드된다. 즉
마크다운의 *별도 단락* 형태 → 한/글의 *인라인 임베드* 형태로 자리를
옮기는 흡수다.

| 요소 | 마크다운 입력 | 흡수 방향 | 한/글 출력 위치 |
|---|---|---|---|
| 표 캡션 | 표 *직전* 단락 | 직전 | `<hp:tbl> > <hp:caption>` |
| 그림 캡션 | 그림 *직후* 단락 | 직후 | `<hp:pic> > <hp:caption>` |
| 각주 본문 | `[^id]: ...` 정의 블록 | id 매칭 | `<hp:footNote>` (본문 안) |

차이점:

- 캡션은 *위치* 기반 매칭 (직전/직후 인접 단락) 인 반면, 각주는 *id 라벨*
  기반 매칭 (`mark.footnote_id == def.footnote_id`).
- 캡션은 paragraph → 표/그림의 메타로 옮기는 단순 이동이지만, 각주는
  paragraph 의 *인라인 마크* (= 본문 안의 특정 offset) 와 정의를 각각 한
  쌍으로 연결한다. walker 가 두 쪽을 dict 매칭으로 짝지어 준다.
- 사용자가 적은 라벨 (`1`, `second` 등) 은 무시되고 markdown-it footnote
  plugin 이 등장 순서로 0, 1, 2 ... id 를 자동 부여한다 (한/글 자동번호와
  같은 철학).
- `inspect` 도구는 본문 단락의 텍스트에서 `<hp:footNote>` 안의 텍스트를
  *제외* 하고 (이중 카운트 방지), 별도의 "각주" 스타일 단락으로 따로
  열거한다. 검증 시퀀스에서 본문 줄과 각주 줄이 깔끔하게 분리되어 보임.

근거 명세는 A 의 `samples/incremental/07_footnote/07_footnote.md` 의
"테스트 관점" 절. 인터페이스 변화는 `mapsi/parser.py`,
`mapsi/ast_walker.py`, `mapsi/builder/elements.py` 의 docstring 에 상세히
기록되어 있다.

## 후속 — 참고문헌 섹션 demote (Phase 8)

캡션/각주가 *블록 → 메타* 흡수였다면, 참고문헌 demote 는 *블록 자체의
역할 재라벨링* 이다. 깊이 1 헤딩의 텍스트가 `참고문헌`, `참고 문헌`,
`References`, `REFERENCES` 의 4 가지 정확 매치 중 하나면, 그 헤딩 *다음*
부터 다음 깊이 1 헤딩이 등장하기 전까지의 `paragraph` /
`bullet_list` / `ordered_list` 의 `role` 을 `reference` 로 (그리고
`depth=0` 으로) 교체한다. 헤딩 자체의 스타일은 보존 (= 기존 매핑인
"개요 1").

| 항목 | 캡션 (표/그림) | 각주 | 참고문헌 |
|---|---|---|---|
| 매칭 키 | 인접 위치 (정규식) | id 라벨 | h1 텍스트 (정확 매치) |
| 변환 형태 | 블록 → 메타 | 블록 → 인라인 마크 | 블록 → 다른 role |
| 영향 범위 | 단일 표/그림 | 단일 paragraph 마크 | 섹션 내 모든 단락/리스트 |
| 헤딩 변경 | — | — | 변경 안 함 (개요 1 유지) |
| 종료 조건 | — | — | 다음 깊이 1 헤딩 |

설계 선택:

- **정규식이 아닌 정확 매치 4 종 고정**. A 명세에 "허용 변형은 위 4 가지로
  고정" 이 명시되어 있어 임의의 변형 (`참 고 문 헌`, `bibliography`) 은
  의도적으로 받지 않는다. 향후 후보를 늘리려면 `REFERENCE_HEADING_TEXTS`
  frozenset 만 손보면 된다.
- **h2 이하 하위 헤딩은 섹션 종료 신호 아님**. 즉 `# 참고문헌 \n ## 국문 자료`
  와 같은 구조에서도 `## 국문 자료` 아래 단락은 여전히 `참고문헌` 스타일.
  새 h1 이 나와야만 섹션이 닫힌다 (단, 그 새 h1 도 다시 후보 텍스트면 그대로
  in_ref 유지).
- **리스트 마크는 reference 로 평탄화**. 사용자가 항목을 `- ` / `1. ` 로
  적었더라도 한/글 출력에서 글머리/번호 없이 평문 단락으로 만든다 (`depth=0`
  + role 교체). 명세의 "참고문헌 섹션 내부에는 글머리 기호/번호가 없어야 함"
  요구사항.
- **표/그림/코드/인용 등 다른 역할은 그대로 통과**. 참고문헌 섹션 안에
  표가 있어도 표 스타일은 유지 (변환 안 함). reference 가 paragraph 류의
  대체 라벨이라는 점에서 자연스러운 경계.

근거 명세는 A 의 `samples/incremental/08_references/08_references.md` 의
"테스트 관점" 절. 구현은 `mapsi/ast_walker.py::_demote_in_reference_section`
한 함수 + `REFERENCE_HEADING_TEXTS` 상수.
